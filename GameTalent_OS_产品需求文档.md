## 一、产品概述与战略定位

### 1.1 产品愿景

GameTalent OS 致力于成为游戏行业最具竞争力的智能招聘管理平台。我们的目标是通过深度融合人工智能技术与游戏行业专属需求，重新定义企业招聘的效率标准与人才获取能力。

我们相信，未来的招聘系统不应仅仅是流程管理工具，而应成为企业的智能人才伙伴。GameTalent OS 通过 AI 技术将HR从繁琐的事务性工作中解放出来，使其能够专注于更具战略价值的人才决策工作。同时，我们通过对游戏行业深刻的理解，为这一垂直领域提供专属的解决方案。

### 1.2 核心价值定位

GameTalent OS 采用「三位一体」的融合架构，集成了行业内领先产品的核心优势，并在此基础上进行创新突破。

**第一，整合 Moka 的智能化体验**。Moka 以其出色的用户界面设计和智能化的简历解析能力著称，是国内HR SaaS领域的标杆产品。我们继承并强化了这一优势，提供消费者级水准的交互体验，让招聘管理工作变得高效且愉悦。

**第二，融入品聘的猎头级管控**。传统 ATS 系统往往缺乏对招聘过程精细化管控的能力，而品聘等猎头管理系统在这一领域积累了丰富经验。我们将这些专业功能引入产品体系，实现从简历筛选到候选人入职的全流程精细化管理。

**第三，构建独有的外部人才 Mapping 能力**。这是 GameTalent OS 的核心差异化竞争力。我们通过分析企业人才库中的历史数据，结合公开情报信息，帮助企业建立对外部人才市场的深度洞察，实现从「被动等待投递」到「主动定向获取」的战略性转变。

### 1.3 目标用户画像

**核心用户群体为 HR 专员与招聘专员**。这一群体是系统的主要使用者，每天需要处理大量的简历筛选、面试协调、入职跟进等工作。他们对系统的主要诉求包括：提升工作效率、减少重复性工作、获得数据支持决策。

**重要用户群体为业务部门负责人（Hiring Manager）**。包括游戏制作人、美术总监、技术总监等岗位。他们参与面试评估，但通常时间有限，希望能够快速了解候选人情况并做出决策。他们期望系统能够提供清晰的候选人评价摘要，以及便捷的移动端操作体验。

**扩展用户群体为外部猎头合作伙伴**。部分企业会与外部猎头合作进行人才招聘，需要为这些外部合作伙伴提供受限的系统访问权限，同时确保企业内部数据的安全性。

**终端用户为求职者**。候选人会通过候选人门户与系统进行交互，查看招聘流程状态、接收面试通知、提交必要材料。他们期望获得良好的求职体验，展现企业专业的雇主品牌形象。

## 二、产品功能架构总览

### 2.1 功能模块全景图

GameTalent OS 由五大核心功能模块组成，各模块之间相互协同，共同构成完整的招聘管理闭环。

**模块一：全域采集与智能解析**。这是整个系统的数据入口，负责从多渠道获取候选人信息，并通过 AI 技术进行结构化处理。核心功能包括浏览器插件一键抓取、全渠道 API 对接、游戏垂直领域技能识别。

**模块二：已入库简历深度挖掘与外部情报 Mapping**。这是产品的战略差异化模块，专注于盘活存量人才资产并建立外部人才情报能力。核心功能包括竞品组织图谱反向推导、存量简历智能激活、人才流向可视化分析。

**模块三：AI 驱动的招聘流程自动化**。负责管理从简历筛选到入职的全流程，通过 AI 技术实现智能化辅助与自动化流转。核心功能包括智能面试排期、流程节点自动化、内部信息权限管控。

**模块四：分级透明的候选人体验中心**。负责向候选人提供招聘流程信息，优化候选人体验。核心功能包括透明度等级配置、内部评价物理隔离、面试官隐私保护。

**模块五：数据决策与薪酬对标**。负责为招聘决策提供数据支持与市场情报。核心功能包括真实成交均价分析、招聘漏斗效能分析、市场薪酬趋势监控。

## 三、模块一：全域采集与智能解析

### 3.1 产品定位与设计理念

模块一的核心目标是构建一个强大、便捷、智能的人才信息采集体系。我们追求「无感捕获」的用户体验——让 HR 在日常工作中自然地积累人才资产，无需额外的操作负担。

传统的招聘系统中，简历采集往往依赖候选人主动投递或 HR 手动录入。这种方式效率低下，且难以获取那些「被动求职者」的信息。GameTalent OS 通过浏览器插件与全渠道 API 对接，实现对全网人才信息的智能采集。

针对游戏行业的特殊性，我们在通用简历解析能力之上，增加了对游戏开发技术栈、美术风格、品类经验等垂直领域知识的深度理解能力。这种垂直化处理使我们的解析准确度远超通用系统。

### 3.2 功能详细设计

#### 3.2.1 浏览器插件「一键抓取」

**功能描述**。为 HR 提供一款浏览器插件，安装后在浏览任何网站时都可以一键抓取候选人信息，并自动导入系统。这借鉴了品聘插件的便捷性，同时增加了智能解析能力。

**目标用户场景**。HR 在浏览 GitHub、ArtStation、LinkedIn、Boss 直聘、牛客等网站时，发现符合企业需求的人才，只需点击插件按钮，即可将该候选人信息导入系统。系统会自动识别页面上的关键信息，包括姓名、头像、当前职位、技能标签、项目经验等，并附带来源链接，方便后续追溯。

**核心功能要求**。

插件应支持主流浏览器（Chrome、Edge、Firefox），安装过程简洁，无需复杂配置。插件启动后会以悬浮小球或侧边栏形式呈现，不影响正常网页浏览。抓取操作支持单条抓取和批量抓取两种模式，适应不同场景需求。系统应自动检测重复候选人，基于手机号、邮箱、姓名等多维度进行去重判断。对于已存在的候选人，应提供「更新」或「新建」两个选项。

**智能解析能力**。抓取过程中，系统应自动提取并结构化以下信息：基本联系方式（姓名、手机、邮箱、LinkedIn 主页）、职业信息（当前职位、公司、工作年限）、技能标签（技术栈、工具软件、美术风格）、项目经验（项目名称、担任职责、业绩描述）、教育背景（学校、学历、专业）、作品集链接（GitHub、ArtStation、 Behance、个人网站）。

**数据同步机制**。插件抓取的数据应实时同步至云端系统，HR 在网页端和客户端都能看到最新采集的候选人信息。同步过程应支持离线缓存，网络恢复后自动上传。

#### 3.2.2 全渠道自动同步

**功能描述**。通过 API 对接主流招聘平台，实现职位发布与简历收取的自动化管理。这一功能继承并强化了 Moka 的多渠道能力。

**支持渠道**。系统应支持主流招聘平台的 API 对接，包括拉勾网、Boss 直聘、猎聘网、智联招聘、前程无忧、脉脉、LinkedIn 等。每个渠道的接入需单独开发适配器，支持配置化开启或关闭。

**自动同步机制**。职位信息应支持一键发布至多个渠道，渠道间的刷新、下线操作应统一管理。简历收取应实现自动下载与解析，无需手动操作。系统应自动进行渠道去重合并，同一候选人从不同渠道投递的简历应归并至统一档案。

**渠道效果分析**。每个渠道的效果应单独统计，包括简历投递量、进入面试数量、最终入职数量、渠道成本等。系统应生成渠道质量分析报表，帮助 HR 优化渠道投放策略。

#### 3.2.3 游戏垂直领域 AI 解析

**功能描述**。针对游戏行业的特殊性，构建专门的 AI 解析模型，能够识别 UE5、Unity、TA（技术美术）等垂直技能，以及游戏品类经验、美术风格等专业维度。

**技术能力要求**。系统应内置游戏行业知识图谱，包含以下核心实体：游戏引擎（Unity、Unreal Engine 4/5、Cocos、LayaBox、白鹭等）、开发语言（C#、C++、Lua、GDScript、GLSL、HLSL 等）、技术美术方向（TA、3D 角色、3D 场景、2D 角色、UI 特效、关卡美术等）、游戏品类（MMO、RPG、SLG、FPS、MOBA、休闲、卡牌、二次元等）、美术风格（写实、卡通、欧美、日系、国风、像素等）。

**解析输出要求**。解析结果应包含结构化的技能标签体系，按「精通」「熟悉」「了解」三个层次分级。系统应自动识别候选人经历中的项目类型，判断其是否有对应品类的开发经验。对于技术岗位，应识别其参与项目的上线状态（已上线、在研、测试中）和项目规模。对于美术岗位，应识别其擅长风格，并从作品集链接中提取样本进行风格分析。

**逻辑疑点识别**。系统应具备简历真实性检测能力，识别以下常见问题：技能描述与项目经历不匹配（如声明精通某引擎但项目经历中未体现）、工作时长异常（如三年经验但罗列了五个大型项目）、项目描述过于笼统缺乏具体细节。系统应在解析报告中标注「需关注」标记，提醒 HR 重点核实。

### 3.3 用户故事

**故事一：HR 在 GitHub 发现潜在候选人**。HR 小王在 GitHub 浏览开源项目时，发现某开发者的代码质量很高，且项目经历与公司正在招聘的 Unity 高级开发工程师岗位高度匹配。小王点击浏览器插件，选择「一键抓取」，系统自动解析了该开发者的 GitHub 主页信息，包括技术栈、项目经历、活跃度等，并附带来源链接。小王随后在系统中创建了针对该候选人的沟通任务，开始主动联系。

**故事二：多渠道简历自动归并**。候选人张同时在 Boss 直聘和拉勾网投递了简历。系统通过手机号匹配，自动将两份简历归并至同一档案，并标注了不同的投递渠道。HR 在查看候选人详情时，可以看到两个渠道的简历内容，以及各自处于不同的招聘流程阶段。

**故事三：游戏美术简历智能解析**。HR 收到了一个美术设计师的简历，除了常规的工作经历外，简历中还包含一个 ArtStation 作品集链接。系统解析时自动识别了该链接，并抓取了作品集的部分缩略图，同时标注了候选人的擅长风格（日系二次元）、使用工具（Photoshop、Blender、CLO3D）、项目经验（女性向手游、欧美卡通）。系统还识别出候选人之前的工作经历与作品集风格存在一定差异，标注为「需关注」。

### 3.4 验收标准

**功能验收标准**。浏览器插件应支持 Chrome、Edge、Firefox 三个主流浏览器，插件安装后可持续稳定运行。插件抓取功能应能在目标网站上正确提取候选人信息，成功率应达到 95% 以上。自动去重功能应能正确识别重复候选人，基于多维度的去重准确率应达到 98% 以上。

**性能验收标准**。单份简历解析时间应控制在 3 秒以内。批量解析 100 份简历应在 5 分钟内完成。系统应支持每日处理 10000 份以上的新增简历。

**解析准确率验收标准**。基础信息（姓名、手机、邮箱）提取准确率应达到 99% 以上。技能标签提取准确率应达到 90% 以上。游戏行业垂直标签提取准确率应达到 85% 以上。

## 四、模块二：已入库简历深度挖掘与外部情报 Mapping

### 4.1 产品定位与设计理念

模块二是 GameTalent OS 的核心战略模块，也是区别于市场上其他招聘系统的关键差异化能力。我们的核心理念是：将企业人才库从「静态存储仓库」转变为「动态 intelligence 资产」。

传统招聘系统中，人才库往往是一个被遗忘的数据墓地——简历被存储起来，但很少被再次利用。GameTalent OS 通过 AI 技术重新激活这些沉睡的简历，同时建立对外部人才市场的深度洞察能力。

我们特别强调「情报」的概念。在游戏行业，人才竞争的核心在于对竞品公司组织架构和人才分布的了解。模块二通过整合内部数据和公开情报，帮助企业构建竞品人才地图，实现精准的定向挖角。

### 4.2 功能详细设计

#### 4.2.1 竞品组织图谱反向推导

**功能描述**。基于库内人才数据，结合公开情报信息，通过 AI 算法反向推导竞品公司的组织架构和关键人才分布。这项功能为企业提供一个「上帝视角」，能够看清目标公司的人才布局。

**数据来源**。系统从以下来源整合数据：库内候选人的历史工作经历（曾经或正在目标公司任职的候选人）、公开信息（LinkedIn 职位信息、公司官网组织架构、新闻报道中的人事变动）、HR 的手动标注（HR 在日常工作中标记的关键人物关系）。

**图谱构建逻辑**。系统应能够识别候选人之间的关联关系，如同一年入职同一公司、同一项目组经历等。基于这些关联关系，AI 算法推导可能的汇报结构和团队边界。系统应支持按公司、事业部、项目组多个层级查看组织图谱。

**可视化呈现**。竞品组织图谱应以节点图形式展示，每个节点代表一个关键人物。节点应显示姓名、职位、加入公司时间等基础信息。节点之间的连线代表可能的汇报关系或同团队关系。HR 可以点击节点查看该人员的详细档案，以及与公司候选人的关联路径。

**关键人识别**。系统应自动识别各公司的关键人物，标注标准包括：在多个候选人履历中被提及、项目负责人或核心成员、具备稀缺技能或丰富经验。关键人信息应支持导出，方便 HR 进行定向联系。

#### 4.2.2 存量简历智能激活（Re-Matching）

**功能描述**。当企业发布新职位时，系统自动扫描人才库中的历史候选人，识别与新职位匹配的候选人，并生成差异化的激活话术。这项功能将人才库变成为一个主动的「人才源」。

**匹配逻辑**。新职位发布后，系统自动提取职位要求的关键技能、经验年限、品类偏好等维度。基于向量相似度算法，在库内历史候选人中寻找匹配度超过阈值的候选人。对于匹配候选人，系统应分析其历史投递记录、面试阶段、流失原因等信息，评估重新触达的可行性。

**激活话术生成**。系统应为每个待激活候选人生成个性化的激活消息。话术应基于候选人历史简历中的亮点内容，与新职位的契合点进行关联。话术风格应差异化，避免让候选人感觉是群发消息。HR 可以选择使用 AI 生成的话术，或手动修改后发送。

**激活效果追踪**。系统应追踪激活消息的发送效果，包括打开率、回复率、进入流程率等指标。基于效果数据，系统应持续优化激活话术的生成策略。HR 应能够在后台查看激活活动的整体效果报表。

#### 4.2.3 人才流向与水源分析

**功能描述**。通过分析库内人才的历史职业轨迹，可视化展示游戏行业的人才流动趋势，帮助企业识别优质「水源」公司和潜在流失风险。

**人才流动热力图**。系统应能够绘制行业人才流动的热力图，展示人才在不同公司之间的流动方向和规模。热力图应支持按时间周期筛选，观察流动趋势的变化。热力图应支持按职能方向筛选（如程序、美术、策划）。

**水源公司识别**。基于人才流动数据，系统应识别哪些公司是行业的「优质水源」——即向目标公司输送人才较多的公司。系统应标注这些公司的特征，如规模、业务类型、发展阶段等。HR 可以重点关注这些公司的员工，作为优先联系对象。

**流失风险预警**。系统应追踪库内候选人的职业动态，当发现候选人可能离职的信号时（如长期未更新简历后突然更新、在职时间接近行业平均周期等），主动提醒 HR。系统还应分析哪些公司的员工流失率较高，帮助企业了解行业人才竞争态势。

**内部人才流动分析**。除了外部人才，系统还应支持分析内部员工的人才流动情况。HR 可以查看员工转入转出各业务线的趋势，识别人员过度集中的风险，为内部人才调配提供数据支持。

### 4.3 用户故事

**故事一：新项目启动时的快速组队**。某游戏公司计划启动一个新的二次元项目，HR 负责在一个月内组建核心团队。HR 在系统中创建了新项目的职位需求，系统自动扫描了人才库，发现有 15 位历史候选人匹配需求，其中 8 位曾经进入过面试流程但因各种原因未入职。系统为每位候选人生成了差异化的激活消息，HR 筛选后向其中 5 位发送了邀请，最终有 2 位进入了面试流程，大大加快了招聘进度。

**故事二：竞品定向挖角**。公司计划从某竞品公司挖角一位资深技术美术。HR 在系统中查看了该公司的组织图谱，发现该技术美术所在团队有 8 人，其中 3 人在公司人才库中有记录。系统显示其中一人曾经在面试后选择其他机会，现已离职。HR 通过这层关系建立了联系，最终成功获取了目标候选人的兴趣。

**故事三：识别行业人才趋势**。HR 负责的市场调研显示，近半年行业内某细分品类（SLG）人才流动频繁。通过系统的人才流动分析功能，HR 发现该公司正在成为该品类的人才输出大户，同时发现几家新晋公司正在大量吸纳该领域人才。这一洞察帮助公司调整了招聘策略，将目标投向这些新兴公司。

### 4.4 验收标准

**功能验收标准**。竞品组织图谱应能够基于库内数据自动构建，支持至少 50 个重点目标公司的图谱生成。存量简历 Re-Matching 应在新职位发布后 1 小时内完成匹配计算。人才流向分析应支持按月/季度/年不同时间维度的数据查看。

**数据准确率验收标准**。组织图谱中人员关系的推导准确率应达到 70% 以上（可接受人工校正）。候选人匹配度评分与人工判断的一致性应达到 80% 以上。

**性能验收标准**。存量简历匹配计算应在 1 万份简历规模下控制在 10 分钟以内完成。页面加载时间应在 3 秒以内。

## 五、模块三：AI 驱动的招聘流程自动化

### 5.1 产品定位与设计理念

模块三的核心目标是在保证招聘质量的前提下，最大程度地减少 HR 的事务性工作。我们追求「流程自驱动」体验——当 HR 完成一个操作后，系统自动推进流程进入下一环节，整个过程流畅无阻。

与传统系统不同，我们将 AI 能力深度嵌入流程的每个环节：AI 辅助简历筛选和评估、AI 协调面试时间并发送邀请、AI 实时辅助面试官、AI 自动生成面试评价。这种全方位 AI 赋能重新定义了招聘效率的标准。

同时，我们借鉴品聘的猎头级管控理念，在流程关键节点设置必要的管控门槛，防止流程失控或信息缺失。

### 5.2 功能详细设计

#### 5.2.1 智能面试排期与协同

**功能描述**。系统自动协调面试官和候选人的时间，生成多个可选时段供选择，一键发送会议邀请和提醒。这项功能将面试安排从一项耗时的工作变为轻松的确认操作。

**日历集成**。系统应支持与主流日历平台集成，包括 Outlook、Google Calendar、飞书日历、企业微信日历等。HR 无需手动查看每个面试官的日程，系统自动获取可用时间。

**时间推荐算法**。系统应基于候选人期望时间、面试官空闲时间、面试时长要求，自动生成 3 个推荐时段。推荐时段应考虑候选人的地理位置，避免安排在深夜或凌晨。如遇面试官日程冲突，系统应自动提出改期建议。

**自动化通知**。面试安排确认后，系统自动生成面试邀请并发送至候选人和面试官。邀请应包含面试形式（视频/现场）、会议链接/地点、时间、面试官信息、注意事项等。系统应在面试前 24 小时、1 小时分别发送提醒。候选人确认参会后，系统应自动更新面试状态。

**变更处理**。如需变更面试时间，系统支持一键重新排期，自动发送新的邀请并取消旧预约。如候选人长时间未响应，系统自动触发催办机制。

#### 5.2.2 流程节点自动化

**功能描述**。基于预设规则，当条件满足时自动触发下一流程节点，减少人工操作。同时对异常情况进行预警，提醒 HR 及时处理。

**自动化规则配置**。系统应支持灵活配置流程自动化规则。规则类型包括：满足条件自动推进（如面试评分超过阈值自动进入下一轮）、超时自动提醒（如候选人停留在某阶段超过设定时间触发预警）、超时自动淘汰（如候选人超过 X 天未响应自动标记为放弃）。

**节点管控**。关键节点应设置必须填写项，HR 必须完成必要信息的填写才能将流程推进至下一阶段。例如，进入 Offer 阶段前必须填写当前薪资、期望薪资、入职日期等关键字段。系统应支持配置不同职位类型的不同流程模板。

**异常预警**。系统应实时监控所有在招职位的流程状态，识别异常情况。异常类型包括：候选人长时间停留在某一阶段、面试官长时间未提交反馈、流程停滞超过预期周期等。系统应通过消息推送、邮件等方式提醒相关 HR。

**流程日志**。所有流程状态变更都应记录详细的操作日志，包括操作人、操作时间、变更内容等信息。日志应支持追溯和审计。

#### 5.2.3 本地化部署与权限管控

**功能描述**。支持私有化部署以满足企业的数据安全要求，同时提供细粒度的权限控制，确保不同角色只能看到应该看到的信息。

**部署方式**。系统应支持公有云、私有云、混合部署多种模式。私有化部署应支持容器化部署方案，简化运维复杂度。数据存储应支持国密加密，满足企业数据合规要求。

**角色权限体系**。系统应内置多种预设角色，包括 HR 管理员、HR 专员、面试官、部门负责人、候选人等。每个角色应配置默认的功能权限和数据权限。权限应支持按组织架构（部门、事业部）、按招聘流程（特定职位）、按数据类型（特定信息字段）多个维度进行配置。

**敏感信息管控**。系统应支持对敏感字段进行特殊保护。敏感字段包括：候选人薪资期望、薪资核算表、内部评价记录、竞品 Mapping 数据等。敏感字段的可见性应根据角色动态调整。例如，普通面试官不可见候选人的薪资期望，HRD 可见全部信息。

**数据隔离**。多租户模式下，各企业的数据应完全隔离。租户间应无法访问任何形式的企业数据。即使是系统管理员，也无法查看具体企业的业务数据。

### 5.3 用户故事

**故事一：一次设置，全程自动**。HR 小李负责一个技术岗位的招聘，在系统中创建了完整的招聘流程模板，设置了自动化规则：初试通过后自动安排复试、复试评分 80 分以上自动进入终面、终面通过后自动触发 Offer 审批流程。在后续的招聘过程中，系统自动完成了大部分流转操作，小李只需要处理异常情况和关键决策，大大减少了工作量。

**故事二：多轮面试的协调难题**。一个高级岗位需要安排四轮面试，涉及 6 位面试官和 1 位候选人。系统自动协调了所有人的日历，生成了多个可行的时间方案。HR 与候选人沟通确认后，系统一键发送了所有四轮面试的邀请，并分别设置了不同会议链接。每位面试官都收到了针对性的面试准备材料。面试结束后，系统自动催办各位面试官提交反馈，并在收集完整后自动汇总生成面试报告。

**故事三：权限管控下的信息隔离**。公司招聘一位技术总监，HR 建立了专项招聘流程。为防止信息泄露，系统设置了特殊权限：只有 HRD 和 CEO 可以查看候选人的薪资期望和背景调查信息，普通 HR 只能看到流程状态和基本简历。面试官的邀请中不显示候选人姓名，仅显示职位。候选人所有内部评价都被隔离存储，候选人端完全不可见。

### 5.4 验收标准

**功能验收标准**。智能排期应能够正确协调至少 10 位面试官和候选人的时间。流程自动化规则应支持配置的数量不少于 50 种。权限配置应支持至少 3 个维度的组合控制。

**自动化率验收标准**。在标准流程下，至少 60% 的流程节点流转应能够自动化完成。异常预警的准确率应达到 85% 以上。

**性能验收标准**。排期计算应在候选人/面试官数量不超过 20 人的情况下 10 秒内完成。流程状态变更应在 1 秒内完成同步。

## 六、模块四：分级透明的候选人体验中心

### 6.1 产品定位与设计理念

模块四专注于优化候选人的求职体验，同时兼顾企业的商业利益和风险控制。我们的核心理念是「拒绝一刀切的透明」——不同类型、不同层级的候选人应获得不同级别的信息透明度。

传统的招聘系统往往采用两种极端：要么对候选人完全不透明（候选人不知道自己的进展），要么完全透明（候选人可以看到所有流程细节）。这两种方式都有其问题：前者会导致候选人体验差、流失率高；后者则会泄露企业内部的招聘策略和决策过程。

GameTalent OS 通过「透明度策略引擎」实现动态、透明度的智能配置。企业可以根据岗位性质、候选人层级、招聘场景灵活选择最合适的透明度等级，实现候选人体验与商业风险的最佳平衡。

### 6.2 功能详细设计

#### 6.2.1 智能透明度等级配置开关

**功能描述**。系统后台提供「透明度策略引擎」，HR 或管理员可针对不同招聘场景一键切换候选人的信息可见范围。

**标准模式（适用于校招、初级/中级社招）**。

此模式面向大众型招聘场景，目标是提升候选人体验和雇主品牌。候选人可见的信息包括：详细的流程节点状态（简历筛选中 → 初试通过 → 复试安排中 → 终试 → Offer 审批中 → 已发放 Offer）、每个节点的下一步引导信息（如「初试通过，请预约复试时间」）、系统推送的面试准备指南和资料（如面试官风格介绍、团队介绍视频）、作品集提交入口、交通指引等辅助信息。

此模式的价值在于：提升候选人的参与感和被尊重感，减少 HR 回答重复问题的工作量，塑造专业、现代化的雇主品牌形象，特别适合需要大量招聘的企业（如校招）。

**极简模式（适用于高管猎聘、核心技术专家、竞品定向挖角）**。

此模式面向高端人才招聘场景，目标是保持招聘的神秘感和企业的谈判主动权。候选人可见的信息仅包括：宏观的评估结果状态（「评估中」「已结束」），不展示任何中间流转节点、无下一步操作提示、隐藏所有敏感信息。

此模式的通知内容极度精简：仅发送必要的时间和地点确认函，最终结果通知（如录用通知或感谢信），不提供任何引导性内容或额外信息。

此模式的价值在于：避免候选人通过进度推断公司的急迫度或内部决策分歧，保护高管薪酬谈判策略，防止候选人与内部人员私下联系，维护高端招聘的私密性。

**自定义模式（适用于特殊项目或特定渠道）**。

此模式面向特殊招聘场景，允许 HR 手动配置对特定候选人开放的字段。HR 可以勾选具体可见的信息项，如开放面试反馈但关闭下一轮面试官姓名，开放进度状态但关闭评估详情。

此模式适用于：特定的高价值候选人、需要特别对待的合作伙伴推荐、内部推荐等场景。

**模式切换规则**。系统应支持在创建职位时预设透明度模式，也支持在招聘过程中动态调整。模式调整应自动应用到该职位下的所有候选人。不同渠道来源的候选人可以预设不同的默认模式。

#### 6.2.2 绝对隔离的内部评价区

**功能描述**。无论采用何种透明度模式，候选人端都应与内部评价信息物理隔离，确保候选人对内部评价内容完全不可见。

**物理隔离机制**。面试官填写的评价、打分、内部备注等信息，应存储在独立的加密区域。候选人门户、候选人通知等面向候选人的所有界面，都不应包含任何内部评价内容。即使是 HR 管理员主动查看候选人信息，也需要专门授权。

**脱敏展示机制**。如果需要向候选人反馈未通过的原因，系统不应直接展示面试官的具体评价内容。系统应提供预设的标准化话术模板（如「岗位匹配度不符」「暂未通过本次招聘流程」等）。HR 可以选择发送预设话术，也可以手动编写脱敏后的反馈内容，但系统应明确提示不得透露具体负面评价。

**数据保护机制**。内部评价数据应与企业业务数据同等级别保护。HR 应无法通过任何方式将内部评价内容导出或转发。系统应记录所有内部评价的访问日志，支持审计追溯。

#### 6.2.3 面试官隐私保护

**功能描述**。在特定模式下，自动隐藏面试官的真实身份信息，防止候选人绕过企业直接联系面试官。

**匿名邀请机制**。在极简模式下，面试邀请仅显示面试官的职位 Title（如「技术委员会成员」「美术总监代表」），不显示真实姓名。面试官的个人头像、LinkedIn 主页等可识别信息都应隐藏。

**沟通隔离机制**。系统应隔离候选人与面试官之间的直接沟通渠道。候选人如果通过邀请中的信息尝试联系面试官，应被引导至系统统一入口。面试官如果主动联系候选人，相关沟通应记录在系统中。

**选择退出机制**。如果企业希望面试官与候选人直接沟通（如为了增强候选人体验），可以手动开启。但所有沟通内容应同步至系统，便于企业掌握完整信息。

### 6.3 场景化应用示例

**场景一：校园招聘**。某游戏公司启动秋季校园招聘，面向 2027 届毕业生招聘 50 个岗位。HR 在系统中预设所有校招职位采用「标准模式」。候选人投递简历后，可以看到完整的进度状态，每一步都有详细的引导信息。当候选人通过初试后，系统自动推送了「复试预约指南」，包含面试形式、常见问题准备、公司介绍视频等。候选人感受到企业的专业和尊重，面试参与率显著提升。

**场景二：竞品工作室主程挖角**。公司计划从竞品公司挖角一位资深主程序，HR 在系统中创建了专项招聘流程，设置为「极简模式」。候选人收到的是「您的申请正在评估中」的状态通知，没有任何进度条，也没有下一步提示。候选人无法通过状态变化判断公司是否在多方对比或者内部是否在犹豫。当公司决定发起 Offer 时，只发送了一封简洁的录用通知。候选人全程不知道公司为其开了多少次内部会议、进行了怎样的薪酬博弈。

**场景三：CTO 级别猎头推荐**。通过猎头推荐一位 CTO 候选人，HR 在系统中将流程设置为「极简模式 + 面试官匿名」。候选人在流程中只能看到「评估中」的状态，以及收到一封包含会议时间和地点的邀请函。邀请函中不显示任何面试官的真实身份信息，仅显示「公司高管代表」。整个招聘过程极其低调，避免任何消息外泄影响现有管理团队。

### 6.4 用户故事

**故事一：校招体验优化**。某游戏公司往年校招候选人流失率很高，候选人普遍反馈「不知道自己的进展」「问了 HR 也没回复」。今年启用 GameTalent OS 的标准模式后，候选人可以随时查看自己的进度，每一步都有系统自动推送的准备材料。HR 发现咨询量下降了 60%，候选人面试参与率提升了 25%。

**故事四：高管招聘保密**。公司通过猎头接触一位竞品公司的 VP，招聘过程需要严格保密。HR 在系统中设置了极简模式，并开启了面试官匿名。候选人全程不知道具体是谁在面试他，只知道是「公司高管」。当招聘进入薪酬谈判阶段，公司内部也只有 CEO 和 HRD 知情。最终候选人顺利入职，整个过程未走漏任何风声。

### 6.5 验收标准

**功能验收标准**。系统应支持三种透明度模式的配置和切换。模式切换应在 1 秒内生效并应用到所有相关候选人。内部评价区的数据应无法通过任何方式泄漏至候选人端。

**候选人可见性验收标准**。标准模式下，候选人应能查看所有定义的可见信息。极简模式下，候选人应完全无法看到任何内部流转节点。自定义模式下，应能精确控制到每个字段的可见性。

**面试官隐私验收标准**。极简模式下，面试邀请中不应包含任何可识别面试官身份的信息。候选人应无法通过系统任何入口获取面试官的联系方式。

## 七、模块五：数据决策与薪酬对标

### 7.1 产品定位与设计理念

模块五的核心目标是将数据转化为招聘决策的支持力量。我们相信，优秀的招聘决策应该基于数据洞察，而非主观判断。

传统招聘系统往往只能提供基础的统计报表，如收到多少简历、安排了多少面试、入职了多少人。这些数据虽然有用，但远远不够。GameTalent OS 提供更深入的数据分析能力，包括招聘漏斗的精细化分析、市场薪酬的实时对标、招聘策略的效果归因等。

我们特别强调「真实成交均价」的概念。市场上的薪酬数据往往来自于候选人的期望或者招聘网站的自报数据，这些数据与实际成交薪酬往往存在较大差距。GameTalent OS 基于企业真实入职数据生成的薪酬分析，具有更高的参考价值。

### 7.2 功能详细设计

#### 7.2.1 真实成交均价分析

**功能描述**。基于企业库内实际入职候选人的薪酬数据，生成各岗位的市场成交价格区间，帮助企业制定更具竞争力的薪酬策略。

**数据来源**。系统应汇总企业所有成功入职候选人的实际薪酬数据。数据应包含基本薪资、绩效奖金、股票期权、签字费等全部薪酬组成部分。数据应按职能、职级、城市、公司规模等维度进行细分。

**统计分析**。系统应生成各岗位的薪酬分布图（Box Plot），展示 P25、P50、P75 分位值。应支持按工作经验年限进行分层分析。应支持与市场公开薪酬数据进行对比。应能识别是否存在薪酬倒挂现象（新员工薪资高于同岗位老员工）。

**趋势分析**。系统应追踪薪酬趋势的变化，按月度/季度展示各岗位薪酬中位数的变化。应能识别特定技能或品类的薪酬溢价趋势。应能预测未来一段时间的薪酬走势。

**数据安全**。所有薪酬数据都应进行加密存储。薪酬报表应支持权限控制，不同级别管理员可见不同粒度的数据。应支持数据脱敏后用于行业对标分析（不暴露具体企业数据）。

#### 7.2.2 招聘漏斗与效能分析

**功能描述**。提供多维度的招聘效能分析，帮助企业识别流程中的瓶颈和优化机会。

**漏斗分析**。系统应展示从职位发布到入职全流程的转化漏斗。漏斗应支持按职位类型、招聘渠道、HR 负责人等维度进行下钻分析。应能识别转化率异常的阶段，并提供预警。

**效率指标**。系统应追踪以下关键指标：平均招聘周期（从职位发布到候选人入职）、简历初筛通过率、各阶段面试通过率、Offer 接受率、招聘成本（按渠道、按职位）、候选人来源分析。

**渠道效果**。系统应分析各招聘渠道的效果，包括简历数量、质量、转化率、成本等。应能识别高效渠道和低效渠道。应能基于数据提出渠道优化建议。

**团队绩效**。应支持按 HR 团队或个人进行绩效排名。应能对比不同 HR 的招聘效率和质量。应能为绩效评估提供数据支持。

#### 7.2.3 期望薪资趋势监控

**功能描述**。追踪候选人的薪资期望变化趋势，识别市场供需变化和薪酬预期走势。

**期望数据收集**。系统应自动收集候选人各轮面试中提到的期望薪资。应能区分期望薪资和最终Offer薪资的差异。应能追踪同一候选人在不同机会下的期望变化。

**趋势分析**。系统应按技能方向、职能类别分析期望薪资的中位数和分布变化。应能与实际成交薪资进行对比，识别期望与实际的差距。应能识别特定技能或品类的期望薪资溢价。

**预警提醒**。当发现特定岗位的期望薪资持续上涨时，系统应自动提醒 HR。可能存在薪酬倒挂风险时，系统应发出预警通知。

**市场情报**。系统应整合公开的市场薪酬报告数据，与内部数据进行对比。应能分析不同城市、不同公司规模的薪酬差异。应能为薪酬策略制定提供市场情报支持。

### 7.3 用户故事

**故事一：薪酬策略优化**。公司在招聘某个关键技术岗位时，多次遇到候选人因薪资期望不符而流失。HR 通过系统的薪酬分析功能，发现该岗位在市场上的成交均价高于公司现行的薪酬带宽。基于此数据，HR 成功向管理层申请调整了该岗位的薪酬范围，后续招聘成功率显著提升。

**故事二：流程瓶颈定位**。公司发现某个岗位的招聘周期特别长，但不清楚问题出在哪个环节。通过漏斗分析，HR 发现该岗位在「终面→Offer」阶段的转化率特别低（仅 20%），而行业平均水平为 40%。深入分析后发现原因是终面通过后薪酬谈判环节耗时过长。基于此发现，HR 改进了薪酬谈判流程，将该阶段的平均周期缩短了一半。

**故事三：渠道效果评估**。公司尝试了一个新的招聘渠道，投放成本较高。三个月后，HR 通过渠道分析功能对比各渠道效果，发现该新渠道的简历数量虽然较多，但转化率极低，最终入职成本是其他渠道的两倍。基于此数据，HR 决定停止该渠道的投放，将预算转移到更高效的渠道。

### 7.4 验收标准

**功能验收标准**。薪酬分析应支持至少 5 个维度的细分（职能、职级、城市、经验年限、公司规模）。漏斗分析应支持至少 10 种维度的下钻。期望薪资趋势应支持按周/月/季不同粒度的查看。

**数据准确率验收标准**。薪酬成交数据的完整率应达到 95% 以上。分析报表的统计口径应清晰可解释，与原始数据的差异率应低于 1%。

**性能验收标准**。常规报表的生成时间应控制在 10 秒以内。复杂分析（如全维度交叉分析）应支持后台异步处理，结果在 5 分钟内生成。

## 八、非功能性需求

### 8.1 性能要求

**系统响应时间**。页面首次加载时间应控制在 3 秒以内。常规交互操作响应时间应控制在 1 秒以内。搜索功能在百万级数据量下响应时间应控制在 2 秒以内。

**并发处理能力**。系统应支持至少 500 用户同时在线。批量操作（如批量发送邮件）应支持队列处理，不影响前端响应。系统应具备自动伸缩能力，应对招聘高峰期（如校招）的流量增长。

**数据处理能力**。系统应支持千万级简历数据的存储和检索。单份简历解析处理时间应控制在 3 秒以内。每日应能处理 10000 份以上的新增简历。

### 8.2 安全要求

**数据加密**。敏感数据应使用 AES-256 算法加密存储。传输过程应使用 TLS 1.3 加密。密钥管理应符合 ISO 27001 标准。

**访问控制**。应实现完善的角色权限体系。应支持多因素认证。应能记录详细的操作日志，支持审计追溯。

**合规要求**。系统应符合 GDPR 等数据保护法规要求。应支持候选人数据的导出和删除请求。应能实现数据保留期限的自动管理。

**渗透测试**。系统上线前应通过第三方安全审计。应能防御 SQL 注入、XSS、CSRF 等常见攻击。应定期进行安全漏洞扫描和修复。

### 8.3 可用性要求

**系统可用性**。系统可用性应达到 99.9%。应支持 7×24 小时运行。年度计划内维护停机时间应不超过 8 小时。

**故障恢复**。应实现数据实时备份，RPO（恢复点目标）应小于 1 小时。应实现服务自动故障转移，RTO（恢复时间目标）应小于 30 分钟。应定期进行灾备演练。

**用户体验**。系统应提供清晰、友好的用户界面。应支持主流浏览器最新两个版本。应提供完善的帮助文档和在线支持。

### 8.4 兼容性要求

**浏览器兼容性**。系统应支持 Chrome、Edge、Firefox、Safari 最新两个桌面版本。移动端应支持 iOS Safari 和 Android Chrome。

**第三方集成**。应与主流日历平台实现同步。应与主流企业通讯工具实现消息互通（飞书、企业微信、钉钉）。应提供标准 API 供第三方系统集成。

## 十、附录

### 10.1 产品路线图

**第一阶段（MVP，1-3 个月）**。核心功能包括：浏览器插件基础抓取、多渠道简历同步、基础简历解析、简单的招聘流程管理、标准模式候选人门户、入职数据薪酬分析。

**第二阶段（4-6 个月）**。增强功能包括：游戏垂直领域深度解析、存量简历智能激活、流程节点自动化、权限管控体系、极简模式候选人门户、招聘漏斗分析。

**第三阶段（7-12 个月）**。高级功能包括：竞品组织图谱反向推导、人才流向分析、智能面试排期、面试官 Copilot、薪酬趋势监控、自定义透明度模式。

**Moka**。国内市场份额领先的 HR SaaS 产品，以优秀的用户体验著称。本产品在智能化体验方面对标 Moka。

**品聘**。专注于猎头行业的招聘管理系统，在流程管控方面有丰富经验。本产品在精细化管控方面借鉴品聘模式。

**Greenhouse**。北美市场领先的 ATS 产品，以灵活的流程配置和强大的协作能力著称。本产品在部分功能设计上参考了 Greenhouse。
